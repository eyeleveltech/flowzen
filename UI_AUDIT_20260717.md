## Design Direction

8 concrete rules to make Flowzen the "simple, clean, readable SaaS app" the brief asks for.

1. **Build one shared `<Table>` primitive and use it everywhere.** Today every list page hand-rolls its own `<table><thead><th>` markup (`components/ui/` has no Table component at all), which is the direct cause of at least two systemic issues below. One primitive = one header style, one row height, one empty-state pattern, forever.

2. **Pick one label-casing convention and encode it in a component, not a class name people have to remember.** Uppercase micro-labels in one place, sentence case in another (Settings vs. Profile) reads as two different products stitched together.

3. **Cap the palette to 4 meanings, not 4+ per screen.** Success/positive, warning, danger, and neutral/info — reused everywhere. The Pipeline board's 10 arbitrary saturated hues is the opposite of the "strict monochromatic palette" the codebase itself says it's aiming for.

4. **Never let text clip without an ellipsis.** Any label that can be user-generated or simply long (module names, client names, task titles) needs `truncate` (or line-clamp), not bare `overflow-hidden`. A hard mid-word clip reads as broken, not busy.

5. **One primary action per screen, visually heavier than everything else.** Most pages already do this well (single black "+ New X" button top-right) — protect that pattern as new pages are added.

6. **Every icon-only control needs a visible label path for touch, not just a hover tooltip.** `title=` attributes are present almost everywhere (good), but they're invisible on tablets/phones. Provide a long-press or always-visible label for destructive/ambiguous icons (e.g. "Cancel document").

7. **Placeholder and hint text must still hit 4.5:1 contrast.** The app-wide `#9CA3AF` gray (used in 42 files) measures ≈2.5:1 on white — it is currently the only visible label on several search inputs, so it isn't merely decorative.

8. **Treat 768px as a real device width, not a rounding error.** A large share of tablets are exactly 768px wide; a strict `< 768` breakpoint check means the compact/mobile layout never activates on the most common tablet size.

## Systemic Issues

Root causes behind multiple individual findings — fix these once, fix everything downstream.

### SYS-1 — No shared Table component → table header styling drifts across the app

**Affects:** 7+ pages

There is no `components/ui/Table.tsx`. Clients, Contracts, Invoices, Subscriptions, Expenses, Departments, and Renewals correctly render headers as uppercase / bold / letter-spaced (the intended style), while Invoice Drafts, Payments, and Receivables render plain Title Case, normal-weight headers — because each page independently wrote its own `<th>` classes and two different conventions leaked in over time.

**Evidence:** `apps/web/src/app/(dashboard)/contracts/page.tsx:49` (`uppercase tracking-wider`) vs. `apps/web/src/app/(dashboard)/invoice-drafts/page.tsx:66-69` and `apps/web/src/app/(dashboard)/payments/page.tsx:40-46` (no uppercase/tracking classes at all).

**Fix:** Extract a `<Table>`/`<Th>` primitive in `components/ui/` that hard-codes `text-xs font-medium text-secondary uppercase tracking-wide`, then migrate every list page onto it. New tables can no longer invent their own style.

Screenshots: `local-1440-contracts.png` · `local-1440-invoice-drafts.png` · `local-1440-payments.png`

### SYS-2 — Sortable column headers silently lose their uppercase styling (browser default overriding Tailwind)

**Affects:** Tasks, Project Detail, Projects, Clients, Pipeline lists

`ColumnDropdown` renders its title text inside a native `<button>`. Browsers apply a default `text-transform: none` user-agent style to `<button>` elements, which overrides the parent `<th>`'s inherited `uppercase` Tailwind class. The result is visible within a single header row: plain columns render uppercase ("TASK", "COMPANY", "ASSIGNEE") while every sortable column right next to them renders mixed case ("Project", "Priority", "Status", "Due Date").

**Evidence:** `apps/web/src/components/ui/column-dropdown.tsx:98-106` — the button renders `{title}` directly with no `uppercase` class of its own. Used by `tasks/page.tsx`, `projects/[id]/page.tsx`, `projects/page.tsx`, `clients/page.tsx`, and `pipeline/components/LeadListView.tsx` (6 files total).

**Fix:** In `column-dropdown.tsx:106` change `{title}` to `<span className="uppercase">{title}</span>` (or add `uppercase` directly to the button's className). One-line fix, five files immediately consistent again.

Screenshot: `local-1440-tasks.png` (look at the header row: TASK/COMPANY/ASSIGNEE are uppercase, Project/Priority/Status/Due Date are not)

### SYS-3 — Pipeline board's 10-color palette contradicts the app's own "strict monochromatic" design system

**Affects:** Primary screen of the CRM module

`PipelineDashboard.tsx` literally comments `// Strict Monochromatic Palette matching design system` and uses greys only for its own charts. Two files away, `PipelineBoardView.tsx` assigns 10 fully-saturated, unrelated hues (violet, cyan, amber, blue, sky, teal, two greens, gray, slate, red) to the 10 pipeline stage columns, then repeats the exact same color as a badge on every card inside the column it's already grouped under — redundant information, delivered loudly.

**Evidence:** `apps/web/src/app/(dashboard)/pipeline/components/PipelineBoardView.tsx:30-55` (`GROUPS` and `STAGE_BADGES` constants).

**Fix:** Reduce to 3–4 semantic colors reused across stages (e.g. neutral = early-stage, blue = active, green = won, red = lost), and drop the per-card stage badge — the column position already conveys the stage.

Screenshot: `local-1440-pipeline.png`

### SYS-4 — Hardcoded `#9CA3AF` used for placeholder/hint text in 42 files, measures ≈2.5:1 contrast (WCAG AA fail)

**Affects:** 42 files, app-wide

See the full contrast measurement in F-1. Because the color is inlined as `text-[#9CA3AF]` per-usage rather than sourced from one token, every one of the 42 usages has to be fixed individually unless a shared low-contrast-safe token is introduced.

**Fix:** Add a `--color-muted` token to `globals.css` (theme block, alongside `--color-secondary`) set to something ≥4.5:1 on white — e.g. `#6B7280` (the existing `--color-secondary`, measured ≈4.84:1) — and do a project-wide replace of `#9CA3AF` → the new token.

### SYS-5 — No consistent label-casing convention between near-identical forms

Settings → Organization uses uppercase micro-labels ("ORGANIZATION NAME", "WEBSITE", "INDUSTRY"); My Profile, one click away, uses sentence case ("Full Name", "Email Address", "Designation (Job Title)") for the same kind of contact-detail fields. No shared `<Field>`/`<Label>` primitive exists, so each page invented its own convention.

**Evidence:** `apps/web/src/app/(dashboard)/settings/page.tsx` (Organization tab) vs. `apps/web/src/app/(dashboard)/profile/page.tsx`.

**Fix:** Standardize on sentence case (more readable at a glance, matches Profile) and build a shared `<FormField label="...">` wrapper so future forms inherit the convention automatically.

Screenshots: `local-1440-settings.png` · `local-1440-profile.png`

## Findings by Theme

### F-1 · P1 · Readability — Placeholder / hint text fails WCAG AA contrast (measured)

**Where:** Global top-nav search ("Search everything..."), every per-page search input ("Search clients...", "Search tasks...", "Search team by name, email..."), and the sidebar "Switch" micro-label.

```
Color: #9CA3AF on #FFFFFF background
Relative luminance (text): 0.3635  |  Relative luminance (bg): 1.0
Contrast ratio = (1.0 + 0.05) / (0.3635 + 0.05) = 2.54 : 1 — FAILS WCAG AA (needs 4.5:1 for normal text)

For comparison, the app's own --color-secondary (#6B7280) on white = 4.84 : 1 — PASSES (barely)
```

**Source:** `apps/web/src/components/layout/top-nav.tsx:144` (global search bar), `apps/web/src/components/layout/sidebar.tsx:122` ("Switch" label, 10px font — even more critical at this size), `apps/web/src/app/(dashboard)/clients/page.tsx:411` (representative of 42 total occurrences of `#9CA3AF` across the codebase).

**Fix:** See SYS-4 — replace with a token that measures ≥4.5:1, e.g. `#6B7280`.

Screenshot: `local-1440-dashboard.png` (top search bar)

### F-2 · P1 · Readability — Module-switcher pill hard-clips "Project Management" without an ellipsis

**Where:** Sidebar module-switcher pill, visible at the top of every PM-module screen (Dashboard, Clients, Projects, Tasks, Calendar, Members, Departments, Reports) at both 1440px and 768px.

**What happens:** "Project Management" renders as "Project Manageme" — cut off mid-word, no "…", reading as a rendering bug rather than a design choice.

**Source:** `apps/web/src/components/layout/sidebar.tsx:117-119` — the label span uses `whitespace-nowrap overflow-hidden` but never sets `text-overflow: ellipsis` (no `truncate` class), while the sibling "SWITCH" badge on line 122 claims fixed space, squeezing the label below its content width.

**Fix:** Replace the classes on the label `<motion.span>` (line 117) with `truncate flex-1 min-w-0` (Tailwind's `truncate` sets `overflow:hidden; text-overflow:ellipsis; white-space:nowrap` in one utility) so long module names ellipsize instead of clipping raw.

Screenshots: `local-1440-dashboard.png` · `local-768-tasks.png`

### F-4 · P3 · Clutter — Unselected filter pills look disabled, not clickable

**Where:** Clients, Tasks, Projects, Calendar and other list-page filter bars ("Status", "Engagements", "Account Manager", "Industries", etc).

**What happens:** Unselected filters use a dashed gray border + muted text (`border-dashed border-gray-300 text-secondary`), which is a common convention for a disabled control, so a first-time user may not register them as active filters worth clicking.

**Source:** `apps/web/src/app/(dashboard)/clients/page.tsx:422` (`triggerClassName` ternary).

**Fix:** Use a solid (not dashed) border at rest — reserve the dashed style exclusively for genuinely disabled controls elsewhere in the app, so the visual language stays a reliable signal.

### F-7 · P0 · States / Responsive — Mobile "More" menu's Sign Out button is functionally unreachable

**Where:** Mobile (390px) — bottom sheet opened from the "More" tab.

**What happens:** The bottom sheet's "Sign out" button and the fixed bottom tab bar are both `position: fixed; bottom: 0` at the same z-index (`z-40`); because the tab bar is rendered later in the DOM it paints on top. Hit-testing confirms it: the Sign Out button occupies y:785.5–836px, the tab bar occupies y:787–844px — a ~49px overlap on a ~50px-tall button. `document.elementFromPoint()` at the button's center returns the tab bar's icon `<svg>`, not the Sign Out button. A tap there activates the tab bar underneath, not sign-out.

**Why this is P0:** There is no other way to sign out from the mobile layout — the desktop sidebar's account/sign-out control isn't present on mobile at all. This blocks a core account action for every mobile user.

**Steps to reproduce:**
1. Open the app at a ≤430px viewport (or any real phone).
2. Tap "More" in the bottom tab bar.
3. Scroll to / observe "Sign out" at the bottom of the sheet.
4. Tap it — the tap registers on the tab bar underneath instead.

**Source:** `apps/web/src/components/layout/bottom-tabs.tsx` — bottom sheet block (lines 92–166, Sign Out at 150-162) vs. Tab Bar block (lines 172–230), both fixed to `bottom: 0` with no bottom padding reserved on the sheet to clear the ~64px tab-bar height.

**Fix:** Add bottom padding to the sheet's scrollable content equal to the tab bar's height (e.g. `pb-[76px]` on the `nav`/Sign Out wrapper at line 151), and/or give the open sheet a higher z-index than the tab bar (e.g. `z-50` vs. `z-40`) so it fully covers rather than sits under it.

Screenshot: `local-390-more-menu-open.png`

### F-8 · P1 · Responsive — Settings tab bar overflows off-screen on mobile with zero scroll affordance

**Where:** Settings page at 390px.

**What happens:** Only 3 of the 9 settings tabs are visible (Organization, Modules, Billing); Users, Workflows, Templates, Notifications, Permissions, and API Keys require a horizontal swipe that nothing on screen hints is possible — no fade/gradient edge, no arrow, no dot indicator, and the thin custom 6px scrollbar (transparent track) is easy to miss on touch where scrollbars are typically hidden entirely.

**Source:** `apps/web/src/app/(dashboard)/settings/page.tsx:104` — `overflow-x-auto` with no visual scroll cue.

**Fix:** Add a right-edge fade/gradient mask (`mask-image: linear-gradient(...)`) whenever the tab strip's content overflows its container, or collapse the tabs into a `<select>` below a set breakpoint.

Screenshot: `local-390-settings.png`

### F-9 · P2 · Responsive — 768px tablet width keeps the full desktop sidebar instead of the compact layout

**Where:** Any authenticated page at exactly 768px (a very common tablet/iPad-portrait width).

**What happens:** The mobile/bottom-tab layout only activates when `window.innerWidth < 768`, so a device that is exactly 768px wide gets the full 260px desktop sidebar, leaving only ~508px for content. Wide tables (Tasks has 7 columns) must then be scrolled horizontally inside that cramped area, and — same root cause as F-8 — there is no visible cue that the table scrolls.

**Source:** `apps/web/src/app/(dashboard)/layout.tsx:30` — `const checkMobile = () => setIsMobile(window.innerWidth < 768);`

**Fix:** Change the comparison to `<= 768` (or better, introduce a real tablet breakpoint around 1024px that collapses the sidebar to icon-only rather than jumping straight to bottom tabs), and add horizontal-scroll shadows to wide tables.

### F-10 · P3 · Responsive / Hierarchy — Module picker grid orphans the third card

**Where:** `/modules` ("Choose a workspace").

**What happens:** A `sm:grid-cols-2` grid holding 3 module cards leaves the third ("Revenue") alone on its own row, left-aligned, with a large empty gap to its right — asymmetric and looks unfinished for a screen that is the very first thing a multi-module user sees after login.

**Source:** `apps/web/src/app/modules/page.tsx:66`.

**Fix:** Use `sm:grid-cols-3` (the cards are compact enough to fit three across at the existing `max-w-3xl` container), or center the last odd card.

### F-11 · P3 · States — Icon-only row actions have no touch-visible label

**Where:** Quotations, Contracts, and similar document tables — a row of 4-5 icon buttons (view/download/duplicate/cancel/delete).

**What happens:** Every icon does carry a `title=` attribute (good — e.g. "View / Edit", "Generate / Download PDF", "Cancel"), but `title` only surfaces on mouse hover. On a tablet or phone there is no hover state, so the meaning of the ambiguous "no-entry" (Ban) icon — is it void, cancel, or disable? — is undiscoverable without tapping it and reading the confirmation dialog that follows.

**Source:** `apps/web/src/app/(dashboard)/quotations/page.tsx:165-176`.

**Fix:** For destructive/ambiguous icons specifically, add a persistent text label at narrow viewports (e.g. swap the icon row for a "⋯" overflow menu with text labels below ~768px), rather than relying on `title` alone.

Screenshot: `local-390-quotations.png`

## Positive Observations

Patterns worth protecting as the app grows.

- **Create/edit drawers (New Task, New Document) are genuinely well designed.** Required fields are clearly starred, related fields are grouped logically (Company/Project, Task Type/Reviewer), and there is exactly one primary action per drawer. This is the pattern the rest of the app should be measured against.

- **Empty states have specific, actionable copy** instead of generic "no data" — e.g. "No invoice drafts yet. Move an accepted quotation to an invoice draft to see it here." tells the user exactly what to do next.

- **Status badges use accessible, semantically-correct color pairs** (`bg-blue-50/text-blue-700`, `bg-green-50/text-green-700`, etc.) with comfortable contrast — this is the right model to extend the Pipeline board's palette toward (see SYS-3).

- **Dashboard information hierarchy is strong:** large KPI numbers at the top, followed by prioritized "Pending/Overdue" lists before secondary charts — the most important number is genuinely the most prominent thing on the page.

- **The mobile "More" bottom sheet** (aside from the Sign Out overlap bug in F-7) is a clean, well-executed pattern — clear icons, logical grouping, smooth spring animation — a good foundation once the z-index/padding issue is fixed.

- **Auth screens (Login/Register/Forgot Password) are simple and uncluttered** — a single centered form, one primary CTA, no competing visual noise. This is close to the "simple, clean SaaS" bar the whole app should meet.
